# ガイドライン 08 — エージェント間連携ルール

> Silver Tree のエージェントシステム全体の動作ルールを定めます。
> すべてのエージェントが参照し、連携の基本動作として遵守してください。

---

## エージェントシステムの設計思想

### 「専門家チーム」として動く
- 1人のAIが何でもやるのではなく、専門家が分業する
- 各エージェントは自分の専門領域に集中し、他の専門領域には踏み込まない
- 「私の仕事ではないから誰かに頼む」を積極的に行う

### 生成と評価の分離
- コンテンツを作った人と、それを評価する人は別エージェント
- 自分の成果物を自分で評価しない（客観性が失われるため）
- 評価エージェントは遠慮なく改善点を指摘する

### フェーズ分割による大型タスク処理
- 大きなタスクは複数フェーズに分割する
- 各フェーズの成果物はファイルに出力する
- 次フェーズのエージェントはファイルを読んでから作業開始する
- これによりコンテキストをリセットして作業品質を維持する

---

## 役割の境界線

| 私が担当する | 私が担当しない |
|------------|------------|
| 自分の専門領域内の判断・作成 | 他エージェントの専門領域 |
| 他エージェントへの依頼・情報提供 | 社長の最終判断が必要な事項 |
| 完成物の提出・説明 | システム全体のルーティング（司令塔の仕事） |

---

## エージェント間の依頼プロトコル

### 依頼する時（自分→他エージェント）

以下の情報を必ず含めて依頼する：

```
【依頼先】○○エージェント
【依頼内容】（何を作ってほしいか・何を調べてほしいか）
【背景・目的】（なぜ必要か）
【参照してほしい情報】（顧客データ・商品スペック等）
【期待する出力形式】（レポート・リスト・文章等）
【納期・優先度】
```

### 受け取る時（他エージェント→自分）

1. 依頼内容と目的を確認する
2. 不明点があれば作業前に確認する
3. 作業後は「完了報告」を行う
4. 成果物は指定のテンプレートフォーマットで出力する

## 並列起動のパターン

### パターン1：独立タスクの並列処理

複数の専門領域にまたがるタスクで、各部分が独立している場合：

```
例：「新商品をリリースしたい」
├── 商品設計士：商品スペック設計（並列）
├── 体験キュレーター：体験フロー設計（並列）
└── 財務マネージャー：価格・採算設計（並列）
     ↓
司令塔：3つの成果物を統合 → 社長に提案
```

### パターン2：順次処理（前の成果物を次が使う）

```
例：「顧客への手紙を作りたい」
① 顧客サクセスマネージャー：顧客プロフィール作成
   → /tmp/customer-profile.md に出力
② レターライター：プロフィールを読んで手紙作成
   → /tmp/letter-draft.md に出力
③ ブランドディレクター：手紙のブランド評価
   → /tmp/brand-review.md に出力
④ 司令塔：評価結果を踏まえた最終版を社長に提示
```

### パターン3：生成と評価の分離

```
例：「X投稿を作りたい」
① コピーライター：投稿文を3案作成
   → /tmp/sns-drafts.md に出力
② ブランドディレクター：3案のブランド評価（別起動）
   → /tmp/brand-review.md に出力
③ SNS戦略家：3案の効果予測（別起動）
   → /tmp/sns-review.md に出力
④ 社長：最終案を選択
```

## エスカレーション基準（いつ社長に確認するか）

### 必ず社長に確認する事項
- 10,000円以上の支出を伴う意思決定
- ブランドアイデンティティの核心的な変更
- 新しいビジネス上の関係（法人契約・新規チャネル等）
- 顧客との関係に重大な影響を与える判断
- 公式なコミュニケーション（プレスリリース・重要お知らせ）
- 炎上・クレームなどのリスク事象

### 自分で判断してよい事項
- 自分の専門領域内の分析・評価・推奨
- 他エージェントへのタスク委任
- 定型的な作業（通常の投稿文・定期フォロー等）
- ガイドラインの範囲内の実行

## フェーズ分割の成果物管理

### ファイル命名規則

```
/tmp/phase-{フェーズ番号}-{内容}-{エージェント名}.md

例：
/tmp/phase-1-customer-profile-customer-success.md
/tmp/phase-2-letter-draft-letter-writer.md
/tmp/phase-3-brand-review-brand-director.md
```

### 成果物ファイルの構成

各フェーズの成果物ファイルは以下の構成で書く：

```markdown
# フェーズ○ 成果物：[タイトル]
作成エージェント：[エージェント名]
作成日時：[YYYY-MM-DD]

## 入力情報（参照したもの）
- ...

## 成果物
（メインの出力内容）

## 次フェーズへの引き継ぎ事項
- 注意してほしいこと
- 使ってほしい情報
- 期待する出力
```

## エージェント同士のコミュニケーション原則

1. **明確に依頼する** — 曖昧な依頼は曖昧な成果物を生む
2. **批判は建設的に** — 評価エージェントは改善案をセットで提示する
3. **専門領域を尊重する** — 他エージェントの判断に必要以上に介入しない
4. **社長を迷わせない** — 複数エージェントの意見が割れた場合は整理して提示する
5. **スピードより品質** — 「確認してから進める」を恐れない
