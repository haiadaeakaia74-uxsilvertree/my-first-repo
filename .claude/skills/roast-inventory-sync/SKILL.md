# /roast-inventory-sync — 焙煎・在庫整合確認

このSkillは、焙煎記録を在庫に反映するときのミスを防ぐために使う。
`roast-log` が焙煎記録とHTML生成を担当し、このSkillは在庫の加算・減算の確認を担当する。

## 原則

- 在庫は勝手に減算しない。まず現在値、使用量、反映済みかを確認する。
- `ops/inventory-log.md` の `## 最新在庫` と `## 焙煎ログ` の両方を見る。
- 在庫の減算は「生豆使用量」で行う。焙煎後重量・歩留まり・販売量とは混同しない。
- 同じ焙煎がすでに記録済みなら、二重減算しない。
- 銘柄名が似ている場合は合算しない。旧豆と新豆を分ける。
- ブレンド用の設計比率と実在庫の減算を混同しない。
- SNS投稿文はこのSkillでは生成しない。

## 減算前チェック

1. 焙煎ログの生豆量を確認する。
2. 対象銘柄が在庫表のどの行か確認する。
3. 既に同じ日付・同じ豆・同じ使用量が `ops/inventory-log.md` にあるか確認する。
4. 減算前後の数値を計算する。
5. 不明点があればユーザーに確認する。

複数バッチの場合:

- 同じ豆なら生豆量を合計してから減算する。
- 同じ日の同じ豆でも、バッチごとのログは残す。
- 在庫ログでは合計使用量と対象バッチが分かるように書く。

プレミックス焙煎の場合:

- 焙煎後に豆別で正確に分離できない。
- 在庫減算は投入時の生豆量で豆別に行う。
- 商品説明では「設計上の内訳」と書き、焙煎後に豆別で分けたように書かない。
- 例: Dark Blendは、ブラジル110g＋コロンビア130gを混合焙煎し、焙煎後の全量を使用する。設計比率としてはブラジル30%、コロンビア35%、コスタリカ35%。

## 入荷時チェック

1. 納品書または写真から銘柄・数量を読む。
2. 既存銘柄に合算するか、新規銘柄として追加するか判断する。
3. 検証豆や産地違いは原則として新規行で分ける。
4. 総在庫を再計算する。
5. 入荷ログに「入荷確認日」を残す。

納品書から読む最低項目:

- 仕入日または納品日
- 正式な商品名
- 数量
- 単位
- 単価
- 税率
- 送料・手数料
- 合計金額

写真が読みにくい場合は、金額や正式名を推測で確定しない。

## 更新対象

- `ops/inventory-log.md`
- 必要に応じて `ops/inventory/index.html`
- 必要に応じて `ops/inventory/YYYY-MM-DD.html`
- 必要に応じて `ops/inventory/green-bean-purchase-plan-YYYY-MM-DD.md`

## 仕入れ優先順位

優先順位は残量だけで決めない。商品への影響で判断する。

1. White / Dark両方に使う共通豆を最優先にする。
2. 商品ラインを止める豆を優先する。
3. 検証用・単品用は次点にする。
4. 似た名前でも切替予定の豆は別行で管理する。

2026-05時点の判断:

- コスタリカ タラス ハニーはWhite / Dark両方に使うため優先度が高い。
- タラス ハニーがなくなり次第、コスタリカ ジャガー ハニー SHBへ切り替える。
- インドネシア系はマンデリン系・深煎り単品ラインを続ける場合の次点。

## 報告フォーマット

```text
在庫反映内容:
- 対象豆:
- 処理: 減算 / 入荷加算 / 確認のみ
- 変更前:
- 変更量:
- 変更後:
- 総在庫:
- 二重反映チェック:
- 次の仕入れ優先:
```

## 禁止

- 生豆量が曖昧なまま減算する。
- 銘柄名が違う豆を合算する。
- すでに反映済みの焙煎をもう一度引く。
- SNS投稿文を自動生成する。
- X投稿、自動投稿、GitHub Actionsに触る。
