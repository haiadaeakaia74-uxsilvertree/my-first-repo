# /roast-log — 焙煎記録シート反映

焙煎記録は `roast-logs/*.md` を正本にする。HTMLは手作業で作らず、原則 `scripts/generate_roast_profiles.py` で自動生成する。

## 基本方針

- ユーザーはコマンドを打たなくてよい。焙煎記録シートを撮影して貼ればよい。
- 写真読み取り → Markdown化 → HTML自動生成 → ブラウザー確認リンク返却、まで行う。
- ROR、歩留まり、重量減少率、焙煎率は手計算よりスクリプト計算を優先する。
- 読み取れない項目を推測で埋めない。曖昧な箇所は「要確認」と書くか、ユーザーに確認する。
- 既存の本番HTMLは無断で上書きしない。まず比較用に生成し、確認後に正式反映する。

## KAKACOO G400 投入量判断

2026-05-16の焙煎記録から、G400では200g以下の少量焙煎が火力・温度上昇の安定を作りにくいと判断する。

- 200g未満はサンプル・方向確認扱いにし、商品用プロファイルの基準にしない。
- 200gは最低ライン。比較に使う場合は、火力ブレの可能性をメモに残す。
- 220〜250gを、G400での記録比較・商品設計の基準投入量にする。
- White / Darkを少量で仕上げたい場合でも、少量焙煎に寄せず、220〜250gで焙煎してから必要量をアフターミックスする。
- 配合比率を検証する間は、豆別焙煎を基本にする。プレミックス焙煎は焙煎後に豆別で分離できないため、比率検証では「設計上の内訳」として扱う。

## 写真から読む項目

最低限、以下を確認する。

- 日付
- 開始時刻
- 豆名
- 用途
- ドラム
- 生豆量
- 焙煎後重量
- 狙い
- 天候 / 気温
- 予熱
- 火力基準
- 0:00からの温度ログ（ドラム温度、豆温、火力、メモ）
- 黄変
- 1ハゼ開始
- 1ハゼ終了
- 2ハゼ
- 火止め
- 排出
- 1ハゼ前後30秒メモ
- 最低限メモ

読みにくい場合は、先に「読めた項目 / 読めない項目」を短く出す。読めない数値を勝手に補完しない。

## 保存手順

1. 写真またはメモから `roast-logs/YYYY-MM-DD_[bean-slug].md` を作る。
2. 温度ログは標準表で保存する。
3. `scripts/generate_roast_profiles.py` で比較用HTMLを生成する。
4. ブラウザー確認用リンクを返す。
5. ユーザー確認後、本番の `roast-profiles/*.html` と `roast-profiles/index.html` に反映する。

標準の比較用生成:

```bash
python3 scripts/generate_roast_profiles.py roast-logs/YYYY-MM-DD_[bean-slug].md --write --overwrite --out-dir roast-profiles/generated-preview
```

正式反映する場合:

```bash
python3 scripts/generate_roast_profiles.py --write --overwrite --index
```

## 自動生成で確認すること

- 個別HTMLに温度グラフが出ているか。
- RORが表示されているか。
- 平均歩留まりが正しいか。
- トータルログに豆別集計が出ているか。
- 豆名表記ゆれで別行になっていないか。
- 「次回確認」「現時点の読み」がMarkdownから拾えているか。

## 在庫反映

在庫減算は `roast-log` の主処理ではない。必要な場合だけ `roast-inventory-sync` または `inventory` の方針で行う。

- 生豆量が明確に読める場合だけ減算候補にする。
- 銘柄が在庫表のどの豆に対応するか確認する。
- 既に減算済みでないか、`ops/inventory-log.md` の最新在庫と焙煎ログを確認する。
- 減算前後の数値をユーザーに分かる形で報告する。

## 禁止

- 写真から読めない数値を推測で埋める。
- 既存HTMLを確認なしに上書きする。
- `roast-logs/*.md` を作らずHTMLだけ作る。
- 在庫を二重減算する。
- SNS投稿文を自動生成する前提にする。
