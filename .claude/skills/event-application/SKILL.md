# /event-application — イベント申込・出店記録スキル

マルシェ・イベントの申込完了・採否・詳細情報を **2ファイル同時に記録** する。
毎回手作業で 2 ファイルを更新する手間をなくす。

---

## Step 1：情報の確認

`$ARGUMENTS` から以下を読み取る。なければ質問する。

| 項目 | 必須 | 例 |
|------|------|-----|
| イベント名 | ○ | ハピマル / 春マルシェ / ○○朝市 |
| 日程 | ○ | 2026/6/28（日）/ 毎月最終日曜 |
| ステータス | ○ | 申込済・審査待ち / 採用 / 不採用 / 詳細確定 |
| 場所 | △ | 高松市○○公園 / 詳細待ち |
| 出店料 | △ | 3,000円 / 無料 / 不明 |
| 区分 | △ | 露店（キッチンカー以外）/ キッチンカー / ブース |
| 備考 | △ | 10:00〜15:00 / 屋外 / 雨天中止 |

---

## Step 2：更新内容を決定

ステータスに応じて更新内容を変える：

| ステータス | strategy-1m.md の更新 | 06-event-operations.md の更新 |
|-----------|---------------------|------------------------------|
| 申込済・審査待ち | 実行計画テーブルに日程追記、タスク①完了マーク | イベントデータ表に新行追加（審査待ち） |
| 採用 | 実行計画の「審査待ち」→「採用・確定」に更新 | 「申込み済み・審査待ち」→「採用・出店確定」に変更 |
| 不採用 | 該当日程を「不採用」マーク → 代替イベントをエントリーするタスク追加 | 備考欄に「不採用」を記録 |
| 詳細確定 | 場所・時間など具体情報を追記 | 「—」「詳細待ち」を実数に更新 |

---

## Step 3：2ファイルを更新

### `strategy-1m.md` の更新箇所

**① 実行計画テーブル**（`## 2本柱の並行実行スケジュール` の `柱A：マルシェ・露店 実行計画`）

該当月行の「アクション」セクションに記載：
```
| 6月〜8月 | 月2回 | ハピマル（6/28・7/26・8/30）申込み済み... |
```

**② 今週やること（該当する場合）**

タスク① `4〜5月の出店先をリストアップ・エントリー` が完了の場合：
```
| ① | ~~...~~ ✅ [イベント名][日程]申込み済み | — |
```

---

### `guidelines/06-event-operations.md` の更新箇所

**「過去のイベントデータ」テーブル** に行を追加：

```
| [イベント名] | [日程] | [場所 or 詳細待ち] | [売上 or —] | [ステータス・備考] |
```

---

## Step 4：次のアクションを提案

更新後、ステータスに応じた次のアクションを提案する：

| ステータス | 次のアクション |
|-----------|--------------|
| 申込済・審査待ち | 採否通知が来たら再度このスキルで更新。準備は採用後に開始 |
| 採用・確定 | 出店1ヶ月前に `/marche-prep [日程] [場所]` で準備開始 |
| 不採用 | 代替イベントを `/sales` または自分でリストアップして再エントリー |
| 詳細確定 | SNS告知文を `/marketing` で作成。採算確認は `/finance` で |

---

## 出力フォーマット

```
## イベント申込記録 — [イベント名] [日程]

### 更新内容
- strategy-1m.md：[更新箇所の説明]
- guidelines/06-event-operations.md：[追加行の内容]

### 次のアクション
- [ステータスに応じた提案]

commit メッセージ案：
📅 [イベント名]出店申込み記録（[日程]）
```
